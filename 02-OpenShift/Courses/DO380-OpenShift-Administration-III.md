## Module 1: Operator Lifecycle Manager (OLM)

OLM helps users install, update, and manage the lifecycle of Kubernetes Operators.

### OLM Objects

- **CatalogSource:** Represents a repository of metadata (CSV, CRDs) describing operators available for install (e.g. Red Hat Operators, Community Operators).
- **OperatorGroup:** Configures which namespaces a selected operator will monitor for custom resources.
- **Subscription:** Keeps the operator updated by defining the installation namespace, update channel (e.g. `stable`), and source catalog.
- **ClusterServiceVersion (CSV):** The manifest representation of a specific operator version, containing permissions, install strategies, and CRD configurations.
- **InstallPlan:** A checklist generated by OLM of resources required to install the CSV. Can require manual or automatic approval.

### Deploying an Operator via CLI

```yaml
# subscription.yaml
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: elasticsearch-operator
  namespace: openshift-operators-redhat
spec:
  channel: stable
  installPlanApproval: Automatic
  name: elasticsearch-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
```
```bash
oc apply -f subscription.yaml

# Verify the CSV succeeds and reaches "Succeeded" phase
oc get csv -n openshift-operators-redhat
```

## Module 2: GitOps Workflows with OpenShift GitOps

OpenShift GitOps uses **Argo CD** to synchronize desired state stored in Git repositories with the actual state running inside OpenShift.

```
  Git YAMLs ──▶ Argo CD (Compares state) ──▶ OpenShift Cluster (Syncs state)
```

### Sync Options
- **`Prune`:** Automatically deletes Kubernetes objects that were removed from the Git repository.
- **`SelfHeal`:** Automatically overwrites manual hotfixes made directly in the cluster with the state declared in Git.

### Argo CD Application Definition
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: production-web
  namespace: openshift-gitops
spec:
  project: default
  source:
    repoURL: https://github.com/winnerineast/gitops-manifests.git
    targetRevision: main
    path: environments/production
  destination:
    server: https://kubernetes.default.svc
    namespace: prod-web
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true
```

## Module 3: User Workload Monitoring

OpenShift provides Prometheus metrics aggregation for cluster services. To scrape custom metrics from your own applications, you must enable **User Workload Monitoring**.

### 1. Enable User Workload Monitoring
Edit the global cluster monitoring configuration ConfigMap:
```bash
oc edit configmap cluster-monitoring-config -n openshift-monitoring
```
```yaml
# Add this data block:
data:
  config.yaml: |
    enableUserWorkload: true
```

### 2. Configure ServiceMonitor
Create a `ServiceMonitor` in your project to instruct Prometheus to scrape your service endpoints:
```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: app-metrics-monitor
  namespace: my-app
spec:
  selector:
    matchLabels:
      app: my-web-app
  endpoints:
  - port: metrics-port # Targets Named Port in the Service definition
    path: /metrics
    interval: 15s
```

## Module 4: Centralized Cluster Logging

OpenShift Logging aggregates stdout/stderr logs from all pods, infrastructure nodes, and system components.
- **Collection Engine:** DaemonSets running Vector or Fluentd collect host logs.
- **LogForwarder:** Filters and routes logs to custom endpoints.

### ClusterLogForwarder Configuration
To ship logs to an external Loki database:
```yaml
apiVersion: logging.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  outputs:
   - name: external-loki
     type: loki
     url: 'https://loki.internal.example.com:3100'
  pipelines:
   - name: forward-app-logs
     inputTypes:
       - application
     outputs:
       - external-loki
```

## Module 5: Node OS Configuration (MachineConfigs)

OpenShift nodes run **Red Hat Enterprise Linux CoreOS (RHCOS)**. Operating system modifications (kernel parameters, host directory configuration, systemd units) are managed via **MachineConfigs** applied by the Machine Config Operator (MCO).

```yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  labels:
    machineconfiguration.openshift.io/role: worker
  name: 99-worker-timezone
spec:
  config:
    ignition:
      version: 3.2.0
    storage:
      files:
      - contents:
          source: data:,UTC
        mode: 420
        overwrite: true
        path: /etc/timezone
```
*Note: Applying a MachineConfig will trigger the MCO to drain, reboot, and reconfigure worker nodes in a rolling fashion.*

## Module 6: Cluster Upgrades

OpenShift clusters require upgrade graph validation using update channels.
- **Fast:** Instant access to releases. Used for testing.
- **Stable:** Delayed releases after extensive soak testing.
- **EUS (Extended Update Support):** Even-numbered releases (e.g. 4.12, 4.14) providing extended lifecycle support.

```bash
# Query current update status
oc get clusterversion

# View update history and available targets
oc describe clusterversion

# Apply cluster upgrade
oc adm upgrade --to-image=quay.io/openshift-release-dev/ocp-release:4.14.10-x86_64
```

## Module 7: Cluster Backup and Restore (OADP)

The OpenShift API Design for Data Protection (OADP) operator backs up cluster state and persistent volumes using Velero.

### Example Backup CRD
```yaml
apiVersion: velero.io/v1
kind: Backup
metadata:
  name: daily-project-backup
  namespace: openshift-adp
spec:
  includedNamespaces:
  - my-app-prod
  storageLocation: default
  volumeSnapshotLocations:
  - default
```

## Module 8: Multi-Cluster Management (ACM)

Red Hat Advanced Cluster Management (ACM) governs multiple OpenShift clusters (spokes) from a single Control Hub.
- **Policy Enforcement:** Define configuration compliance policies (e.g. enforce RBAC or NetworkPolicies) and push them cluster-wide.
- **Placement Rules:** Target applications dynamically to spoke clusters based on cloud labels (e.g. `cloud=aws`, `environment=production`).

## Practice Lab: Setting Up User Workload Monitoring

### Objective
Enable user workload monitoring, deploy a Python app emitting prometheus metrics, and configure a ServiceMonitor.

### Step 1: Enable Workload Scraping
```bash
# Apply monitoring config to openshift-monitoring namespace
cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
  namespace: openshift-monitoring
data:
  config.yaml: |
    enableUserWorkload: true
EOF
```

### Step 2: Create Test Project & Deploy App
```bash
oc new-project metrics-test

# Deploy application that exposes prometheus metrics on port 8080 at /metrics
oc create deployment promo-app --image=quay.io/brancz/prometheus-example-app:v0.1.0
oc expose deployment/promo-app --port=8080 --name=promo-svc
```

### Step 3: Configure ServiceMonitor
```yaml
# service-monitor.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: promo-monitor
  namespace: metrics-test
spec:
  selector:
    matchLabels:
      app: promo-app
  endpoints:
  - port: 8080
    path: /metrics
    interval: 10s
```
```bash
oc apply -f service-monitor.yaml
```

### Step 4: Verify Scraping
Log into the OpenShift Web Console under Developer Mode -> Monitoring -> Metrics, and search for the query `http_requests_total`. You should see live metric graphs aggregated from the Python container.
